# Phase 0에서 50k RPS 생성 경로를 증명한 과정

Phase 0의 목적은 애플리케이션 성능 측정이 아니었다. 먼저 부하 생성기와 ALB만으로 목표 트래픽을 만들 수 있는지 확인해야 했다. collector나 Kafka를 추가하기 전에 측정 장비의 한계부터 걷어내려고 경로를 `부하 생성기 -> ALB 고정 응답`으로 제한했다.

최종 구성은 `1 x c6in.large`의 2 vCPU 생성기였다. SDK 호환 페이로드를 무작위로 바꾸면서 연결 10,000개와 약 55k RPS를 유지했다. 그전까지 낮은 처리량이 반복됐지만 이후 구성 변경으로 계정 할당량, Spot 배치, 반복되는 연결 생성, Artillery 프로세스 구조, 무제한 발사 방식이 ALB 앞단의 측정을 제한했음을 확인했다. 여러 조건을 함께 바꾼 실행에서는 각 조건의 기여도를 따로 계산하지 않았다.

## 먼저 읽을 결론

먼저 결과부터 보면, 초기 실험에서 보인 500~640 RPS는 ALB의 고유 한계가 아니었다. 이후 생성 구성을 바꾸자 같은 고정 응답 구조의 ALB가 완전한 60초 구간에서 595k RPS까지 응답했다. 초기 생성기와 연결 경로가 제한 요소였다는 사실은 확인했지만 세부 원인의 기여도까지 모두 분리한 것은 아니다.

최종적으로 사용한 구성은 다음과 같다.

- 생성기: `1 x c6in.large`, 2 vCPU
- 요청: 실제 SDK 형식과 호환되는 1,047~1,526바이트 요청 본문 12개 중 하나를 매번 무작위 선택
- 연결: HTTP 지속 연결(keep-alive) 10,000개
- 준비 구간: 20k RPS로 30초 예열(warm-up)
- 측정 구간: 목표보다 10% 높은 55k RPS로 150초
- 결과: ALB 완전 60초 구간 `54,999.08 RPS`, 요청 오류 0

인스턴스를 계속 키워서 해결한 게 아니다. 요청마다 새 TCP 연결을 만드는 비용을 줄였고 Node 기반 생성기의 단일 프로세스 한계를 따로 확인했다. 인프라 한계 측정에는 더 가벼운 oha를 썼다. 측정 전에 연결 경로를 예열하고 생성률도 목표 수준으로 제한했다.

Phase 0에서 얻은 결론은 단순한 ALB 한계 수치가 아니다. 부하 생성기가 실제로 목표 RPS를 만드는지 먼저 증명해야 이후 단계의 병목도 제대로 해석된다.

## 프로젝트를 처음 보는 독자를 위한 배경

LoopAd의 이벤트 요청은 원래 ALB 뒤의 collector를 거쳐 Kafka로 전달된다. Phase 0에서는 collector와 Kafka를 의도적으로 빼고 ALB가 즉시 204를 반환하게 했다. 요청 생성 구간만 따로 재기 위해서다.

LoopAd에서 브라우저나 SDK가 이벤트 하나를 전송하면 최종 시스템에서는 대략 다음 경로를 지난다.

```text
SDK
  -> ALB
  -> event collector
  -> Kafka
  -> 후속 저장소와 분석 시스템
```

이 전체 경로에 바로 50k RPS를 보내 실패하면 원인이 너무 많다. SDK 형식 변환, 네트워크, ALB, collector CPU, Kafka producer, broker 처리량 중 어디가 막혔는지 한 번에 구분하기 어렵다.

Phase 0에서는 ALB 리스너가 특정 경로에서 대상 서비스를 호출하지 않고 HTTP 204를 직접 반환하도록 했다.

```text
부하 생성기
  -> SDK 호환 이벤트 하나를 담은 HTTP POST
  -> 내부 ALB
  -> 고정 HTTP 204 응답

제외: event collector, Kafka, ClickHouse
```

이 경로의 204는 이벤트 저장을 뜻하지 않는다. ALB가 요청을 받아 고정 응답을 만들었다는 표시다. Phase 0의 성공만으로 collector나 Kafka 성능을 보장할 수는 없다. 다만 이후 단계에서 처리량이 떨어졌을 때 생성기와 ALB 앞단은 이미 목표를 만들었다고 판단할 근거가 생긴다.

부하 생성기는 테스트 대상 밖의 단순한 도구처럼 보이지만 실제로는 고성능 네트워크 프로그램이다. 정해진 시간에 요청을 예약하고 JSON 본문을 준비한다. TCP 소켓을 열거나 재사용하고 응답을 읽으며 시간 초과와 통계도 관리한다. 이 프로그램이 먼저 느려지면 ALB에 여유가 있어도 낮은 RPS만 보인다.

Phase 0의 측정 대상은 ALB 하나가 아니라 `부하 생성기 -> 네트워크 연결 -> ALB 고정 응답` 전체다. collector와 Kafka는 이 단계의 결과에서 의도적으로 제외했다.

## 지표와 용어를 읽는 방법

RPS, 연결 수, 동시에 처리 중인 요청 수는 서로 다른 값이다. 이 차이를 놓치면 연결이 많아질 때 생기는 시간 초과를 서버 처리량 부족으로 잘못 해석하기 쉽다.

| 용어 | 이 문서에서의 의미 |
| --- | --- |
| RPS | 1초 동안 실제로 전송하거나 완료한 HTTP 요청 수. 문맥에 따라 생성기 수치와 ALB 수치를 구분해서 읽는다. |
| 연결 | TCP 연결 하나. keep-alive를 사용하면 한 연결이 시간 순서대로 여러 HTTP 요청을 처리한다. |
| 동시 연결 | 같은 시점에 열려 있는 연결 수. RPS와 일대일 대응하지 않는다. |
| 처리 중 요청 | 전송을 시작했지만 아직 응답이나 시간 초과를 받지 않은 요청. 열린 연결 수와도 다르다. |
| 지연시간 | 요청 전송부터 응답까지 걸린 시간. p95 이내에 전체 요청의 95%가 끝난다. |
| 시간 초과 | 정해진 시간 안에 연결 또는 응답이 완료되지 않았다는 클라이언트 관측. 서버가 명시적으로 거절했다는 뜻은 아니다. |
| VU | Artillery가 표현하는 가상 사용자 흐름. VU 수를 늘려도 같은 프로세스의 이벤트 루프를 공유한다. |
| 고정 응답 건수 | ALB가 직접 204를 만든 요청 수. 생성기 완료 건수와 대조해 요청이 ALB까지 도달했는지 확인한다. |

최종 실행은 연결 10,000개로 약 55k RPS를 만들었다. 단순 평균으로 보면 연결 하나가 초당 약 5.5개 요청을 처리한 셈이다. 연결 10,000개가 매초 한 번씩만 요청한다는 뜻도 아니고 동시에 요청 10,000개가 항상 처리 중이라는 뜻도 아니다.

ALB `ActiveConnectionCount`도 생성기 설정값을 그대로 복사한 숫자는 아니다. CloudWatch가 1분 구간에서 여러 ALB 노드가 보고한 값을 `Sum`으로 집계하므로 연결 재수립과 집계 시점의 영향을 받는다. 이 실험에서는 정확히 10,000이어야 하는 등식으로 보지 않았다. 목표 규모의 연결이 실제 ALB까지 형성됐는지 확인하는 하한으로 썼다.

동시에 처리 중인 요청 수는 대략 다음 관계로 근사한다.

```text
처리 중 요청 수 ≈ 초당 요청 수 x 평균 응답시간(초)
```

예를 들어 50k RPS에서 평균 응답시간이 20ms라면 약 1,000개 요청이 응답을 기다린다. 응답시간이 200ms로 늘면 같은 RPS에서도 약 10,000개가 대기한다. 응답 지연이 길어질수록 대기 요청은 소켓과 버퍼를 오래 점유하고 그 여파로 응답이 다시 느려질 수 있다. 과부하가 시작된 뒤 시간 초과가 급격히 늘어나는 이유 중 하나다.

### TCP 연결을 새로 만들 때 생기는 일

HTTP 요청 전에 TCP 연결이 없다면 클라이언트와 서버는 먼저 핸드셰이크를 수행한다. 생성기 운영체제는 소켓과 파일 디스크립터를 할당하고 출발지 IP와 임시 포트를 선택한다. 파일 디스크립터는 열린 소켓 같은 운영체제 자원을 가리키는 번호다. 프로세스와 시스템마다 허용된 수가 제한된다. ALB 쪽도 연결 상태를 보관한다. 연결을 닫은 뒤에도 운영체제는 늦게 도착한 패킷이 새 연결과 섞이지 않도록 일정 시간 상태를 유지하기도 한다. 이 상태가 흔히 `TIME_WAIT`로 보인다.

낮은 RPS에서는 이 비용이 작아 보인다. 하지만 요청마다 연결을 새로 만들면 초당 수천 번의 핸드셰이크, 소켓 할당, 종료 처리가 반복된다. 이를 연결 교체(connection churn)라고 한다. 한 출발지 IP의 임시 포트와 커널 자료구조도 유한하다. 아직 처리하지 못한 연결이 기다리는 수신 대기열(backlog)이 길어지거나 스케줄링이 늦어지면 애플리케이션 CPU와 네트워크 대역폭이 남아 있어도 새 연결을 제때 만들지 못한다.

HTTP keep-alive는 이미 만든 TCP 연결에 다음 HTTP 요청을 보내 이 비용을 줄인다. 목적은 응답시간 몇 ms를 줄이는 데 그치지 않는다. 고RPS에서 핸드셰이크와 소켓 생명주기 관리를 요청 수만큼 반복하지 않게 한다.

### 생성기와 ALB 중 어디가 막혔는지 구분하는 방법

생성기 지표 하나만 보면 요청을 보내려 한 건수와 ALB가 실제로 받은 건수가 섞인다. 그래서 다음 수치를 함께 봤다.

1. 생성기가 계획하거나 시도한 요청 수
2. 생성기가 204 응답까지 완료한 요청 수
3. ALB `HTTP_Fixed_Response_Count`
4. ALB 연결 거부와 ELB 5xx
5. 생성기 CPU, 네트워크, 소켓, 시간 초과

생성기 완료 건수와 ALB 고정 응답 건수가 같으면 요청이 ALB에 도달해 응답까지 돌아왔다는 강한 증거다. 반대로 생성기는 많은 요청을 예약했는데 ALB 건수가 작고 ALB reject도 없다면 생성기 내부 대기열이나 연결 수립 단계부터 살펴야 한다.

CPU도 같은 방식으로 해석한다. 16 vCPU 호스트에서 단일 스레드 하나가 100%여도 호스트 전체 평균은 약 6.25%로 보인다. 전체 CPU가 낮다고 모든 실행 경로에 여유가 있지는 않다.

고부하 결과는 RPS 하나로 판정하지 않는다. 연결 생명주기, 처리 중 요청, 생성기와 ALB 양쪽의 건수, 시간 초과가 발생한 위치, 스레드나 이벤트 루프 단위의 포화를 함께 본다.

## 목표와 완료 기준

목표는 순간적으로 50k 요청을 예약하는 데 있지 않았다. 실제 SDK와 비슷한 요청을 ALB까지 보내고 60초 이상 안정적으로 응답받아야 했다. 처리량뿐 아니라 연결 수, 성공률, 양쪽 건수의 일치 여부도 완료 기준에 넣었다.

최종 목표는 HTTP 요청 하나에 이벤트 하나를 담아 초당 50,000개 이벤트를 보내는 것이었다. 요청 본문은 실제 SDK envelope와 호환되고 약 1.0~1.5 KiB 범위여야 했다.

Phase 0을 완료하기 위해 사용한 기준은 다음과 같다.

- 완전한 60초 ALB 구간에서 `50,000 RPS` 이상
- ALB `ActiveConnectionCount` 합계 `10,000` 이상
- HTTP 204 성공률 `99.9%` 이상
- 생성기 완료 건수와 ALB 고정 응답 건수 차이 `2%` 이하
- ALB 연결 거부와 ELB 5xx가 없어야 함

이 기준은 ALB가 collector 없이 트래픽을 받을 수 있다는 뜻일 뿐이다. collector 처리량, Kafka 게시 지연시간, 메시지 유실 여부는 Phase 1 이후의 문제다.

완전한 60초 구간을 요구한 이유는 짧은 순간 부하가 지속 가능한 처리량처럼 보이는 것을 막기 위해서다. 성공률과 건수 차이도 함께 봤다. 생성기가 50k 작업을 예약했어도 실제 ALB에 훨씬 적게 도달했다면 성공으로 볼 수 없기 때문이다.

Phase 0은 `지속 처리량 + 연결 유지 + 오류율 + 양쪽 건수 일치`가 동시에 성립해야 성공이다. 하나라도 빠지면 50k RPS를 증명했다고 보지 않았다.

## 진행 과정 요약

실험은 인스턴스를 계속 키우는 방식으로 진행하지 않았다. 페이로드 생성, 연결 재사용, 출발지 분산, 프로세스 병렬성, 도구 선택, 예열, QPS 제한을 바꾸면서 실패 후보를 좁혔다.

모든 실행이 변수 하나만 바꾼 대조 실험은 아니었다. 한 조건만 바꾼 실행은 그 변수의 효과 비교가 가능하지만 여러 조건을 함께 바꾼 실행은 새 구성 전체가 성공하거나 실패했다는 사실만 보여준다. 아래 표와 본문은 직접 측정한 값, 여러 조건을 묶은 구성 비교, 측정값에 근거한 추론을 구분해서 적는다.

| 시도 | 관측 | 해석 범위 | 다음에 바꾼 것 |
| --- | --- | --- | --- |
| Fargate 소규모 확인 | 10 RPS, 300/300 성공 | capacity provider와 로그 권한이 먼저 필요했음 | 실행 환경 권한과 정리 절차 확립 |
| Fargate 50k/17.5k | 트래픽 시작 전 실패 | 80 Spot vCPU 할당량 초과, 할당량 안에서는 Spot 용량 부족 | 더 작은 목표와 On-Demand 검토 |
| Fargate 5k/10k 및 단계별 증가 | 단계별 증가에서 약 643 RPS부터 시간 초과 | ALB와 생성기의 연결 처리 중 어디가 원인인지 불명확 | 사전 직렬화로 페이로드 비용부터 확인 |

관측 처리량이 낮을 때마다 서버 용량부터 늘리지는 않았다. 가능하면 한 변수를 고정해 비교했고 여러 변수를 함께 바꾼 경우에는 개별 원인을 확정하지 않고 구성 단위의 결과로 남겼다.

## 첫 시도는 ALB 성능 실험까지 가지 못했다

Fargate 작업 컨테이너가 계정 할당량이나 Spot 용량 문제로 시작되지 않았다면 ALB 성능을 잰 것이 아니다. 부하를 보낼 컴퓨터가 실제로 실행됐는지부터 성능 결과와 분리했다.

Fargate는 서버를 직접 관리하지 않고 컨테이너 단위로 실행하는 AWS 컴퓨팅 방식이다. Fargate Spot은 남는 AWS 용량을 저렴하게 쓰는 대신 필요한 시점에 자리가 없기도 한다. 계정 할당량(quota)은 사용자별 요청 상한이고 Spot 용량은 그 순간 AWS에 실제로 남은 자원이다. 할당량 안에 들어와도 작업 컨테이너 배치를 보장하지 않는다.

첫 소규모 확인도 한 번에 성공하지 않았다. 빈 Artillery 인자가 전달된 로컬 실행, `FARGATE_SPOT` capacity provider 누락, CloudWatch Logs 권한 누락을 차례로 고친 뒤에야 Fargate Spot 작업 컨테이너 하나가 10 RPS를 보냈고 300개 요청이 모두 204로 끝났다. 앞선 세 실패는 요청이 시작되기 전의 실험 환경 문제로 분류했다.

50k 시도는 작업 컨테이너 20개에 총 80 Spot vCPU를 요구했지만 계정 할당량은 30 vCPU였다. 목표를 17.5k와 28 vCPU로 줄여 할당량 안에 넣어도 Spot 용량 부족으로 배치되지 않았다. 두 실행 모두 ALB에 요청이 도달하지 않았으므로 ALB 처리량을 판단할 자료가 아니다.

계정 할당량을 통과하고 실행 용량을 확보한 것만으로는 성능 실험에 성공했다고 볼 수 없다. 이후 재현성이 필요한 실행은 On-Demand 또는 EC2 생성기를 사용했다.

로그 권한 문제도 같은 범주다. 작업 컨테이너가 시작 전에 로그 초기화에서 실패하면 요청은 한 건도 생성되지 않는다. 이후 기록에서는 `트래픽 시작 전 실패`와 `트래픽 도중 성능 실패`를 구분했다.

반복 가능한 고부하 측정에서는 저렴해도 언제 부족해질지 모르는 자원보다 실행 여부가 예측되는 생성기 용량이 우선이었다. Fargate Spot 실패 수치는 ALB 한계 계산에서 제외했다.

## 500~640 RPS에서 막힌 이유

시간 초과가 많다는 이유만으로 ALB가 요청을 거절했다고 볼 수는 없다. 이 단계의 지표는 생성 호스트와 TCP 연결 경로를 먼저 의심하게 했지만 정확한 병목 위치까지 측정한 것은 아니었다.

소켓 시간 초과는 생성기가 제한 시간 안에 연결이나 응답을 마치지 못한 상태다. 서버가 느릴 때도 생기지만 생성기 이벤트 루프가 제때 소켓을 다루지 못하거나 새 TCP 연결이 수신 대기열에서 기다릴 때도 같은 오류가 난다. 오류 문자열만 보고 병목 위치를 확정하지 않은 이유다.

Fargate에서 트래픽이 실제로 시작된 5k와 10k 시도는 대부분 소켓 시간 초과로 끝났다. 부하를 단계별로 올린 실행에서는 약 561 RPS까지 오류가 없었고 약 643 RPS부터 시간 초과가 나타났다. 당시에는 ALB, 네트워크, Artillery 연결 대기열 중 무엇이 원인인지 분리하지 못했다.

요청마다 JSON을 구성하는 비용부터 제거했다. 첫 사전 직렬화 실행은 로컬 생성기가 외부 경로로 ALB에 접근했고 Fargate 재시도에는 S3 버킷, 자격 증명, 경로 조건 문제가 섞였다. 이 결과는 내부 경로 성능 수치에서 제외했다.

이후 내부 경로와 Fargate On-Demand 조건을 복구하자 1 vCPU 작업 컨테이너 20개가 모두 시작됐다. 이 실행도 19~22k RPS 부근에서 시간 초과가 급증했고 단일 4 vCPU 작업 컨테이너의 단계별 증가 실행에서는 약 639 RPS부터 시간 초과가 나타났다. 앞선 실행과 모든 조건을 고정한 비교는 아니므로 페이로드 생성 비용의 크기까지 계산할 수는 없다. 다만 작업 배치와 JSON 생성을 해결해도 실패가 남았으므로 둘 중 하나가 유일한 원인은 아니었다.

다음 단일 EC2 단계별 부하에서도 500 RPS는 성공했지만 4,000 RPS 성공률은 1.2%였다. ALB 연결 거부는 없었고 네트워크 사용량도 작았지만 EC2 CPU 최대 평균은 약 79.0%였다. 한 출발지 IP에서 짧은 연결을 계속 만들고 여러 Node 프로세스를 한 호스트에 올린 구성이라서 호스트 CPU, 연결 교체, 출발지 집중의 영향은 분리되지 않았다. 확인된 사실은 이 구성으로 500 RPS 이후의 유효 부하를 만들지 못했다는 점이다.

다음에는 생성기를 4대로 나눠 출발지를 분산하고 keep-alive를 함께 적용했다. 호스트 수와 연결 재사용을 동시에 바꾼 비교였다.

생성기를 여러 호스트로 나누면 CPU만 늘어나는 게 아니다. 출발지 IP, 임시 포트 공간, 커널 소켓 대기열도 함께 나뉜다. keep-alive는 각 호스트가 이미 만든 연결을 재사용하게 한다. 두 변경을 함께 적용하면 요청마다 새 연결을 만들고 버리는 일이 크게 줄어든다.

미리 만든 페이로드만으로는 실패가 사라지지 않았다. 출발지 분산과 keep-alive를 함께 적용한 다음 실행에서는 처리량이 15.6k RPS까지 올라갔다. 어느 변경이 얼마나 기여했는지는 분리되지 않았지만 초기 500~640 RPS가 ALB의 고유 한계가 아니라 당시 생성 구성에 종속된 값이라는 점은 확인됐다.

## 낮은 CPU와 Artillery 프로세스 포화

VU 수와 실제 병렬 실행 단위는 다르다. 한 Node 프로세스 안에서 VU를 늘려도 모두 같은 이벤트 루프를 쓴다. 이 때문에 호스트 CPU 사용률은 낮은데 프로세스는 포화되는 현상이 생길 수 있다.

Artillery는 Node.js 위에서 실행된다. Node는 네트워크 I/O를 비동기로 처리하므로 한 스레드가 여러 소켓을 기다린다. 하지만 요청 예약, 타이머 실행, 콜백 처리, 요청 본문 준비, 지표 집계 같은 JavaScript 작업은 기본적으로 하나의 이벤트 루프가 차례로 처리한다. 비동기 처리에서는 대기하는 동안 다른 일을 한다. 한 프로세스가 소켓 작업을 무한히 동시에 처리한다는 뜻은 아니다.

VU는 독립 사용자의 행동 흐름을 나타내지만 VU마다 운영체제 프로세스나 CPU 스레드가 생기지는 않는다. 같은 Artillery 프로세스에 속한 VU는 하나의 이벤트 루프에 작업을 추가한다. 이벤트 루프가 이미 밀린 상태라면 VU를 더 늘려도 실행 통로는 늘지 않고 대기열만 길어진다.

EC2 생성기 4대로 출발지를 분산하고 각 keep-alive VU가 같은 연결을 반복 사용하도록 바꿨다. 호스트마다 Artillery 프로세스를 하나만 실행했을 때 요청 750만 개는 모두 성공했지만 최대 처리량은 15.6k RPS였다. 호스트 CPU 평균 최대치는 16.71%에 불과했다. 더 큰 인스턴스가 필요한 것이 아니라 단일 Node/Artillery 프로세스가 먼저 포화된 상황이었다.

다음 실행은 호스트 4대, 총 VU 10,000개, VU당 요청 750개, 대기시간 0.15초를 유지했다. 달라진 핵심은 호스트당 VU 2,500개를 프로세스 하나에 넣던 구성을 프로세스 4개와 프로세스당 VU 625개로 나눈 것이다. 최대 처리량은 15,612.85 RPS에서 49,305.75 RPS로 3.16배 올랐고 EC2 CPU 최대 평균은 16.71%에서 21.52%로 변했다. 별도 실행 사이의 비교라는 한계는 있지만 이 구간은 프로세스 분할의 효과를 비교적 직접적으로 보여준다.

그 상태에서 프로세스 수를 유지하고 VU만 675개로 8% 늘리자 처리량은 49,534.83 RPS로 0.46% 증가했다. 반면 EC2 CPU 최대 평균은 21.52%에서 47.10%로 올랐다. 처리량은 소폭 증가했지만 투입한 VU와 CPU 사용량에 비해 한계효과는 작았다.

다음 실행에서는 호스트당 프로세스를 5개로 늘리고 프로세스당 VU를 625개로 되돌렸다. 이 구성은 완전한 60초 구간에서 50,368.63 RPS를 기록해 처음으로 목표를 통과했다. 다만 이전 실행과 비교하면 프로세스 수뿐 아니라 총 VU도 10,800개에서 12,500개로 15.7% 늘었다. 시간 초과도 1,422개 남았다. 목표를 통과한 구성이라는 사실은 확인했다. 그러나 처리량 증가를 독립 이벤트 루프 하나의 효과로만 단정하기는 어렵다.

프로세스마다 별도의 이벤트 루프가 생기므로 프로세스 분산이 처리량 증가에 기여했을 가능성은 있다. 그러나 이 세 실행은 프로세스 수와 총 VU를 동시에 바꿨고 각 조건을 한 번씩만 측정했다. 여기서 확인되는 범위는 4프로세스 구성에서 VU 추가의 한계효과가 작다는 점과 5프로세스 구성이 처음으로 50k 기준을 통과했다는 사실이다. 프로세스 수의 효과를 분리하려면 호스트, 총 VU, 입력 부하, 요청 수와 측정 시간을 고정한 채 프로세스 수만 바꾼 반복 측정이 필요하다.

## Artillery와 oha가 답하는 질문은 달랐다

부하 테스트 도구는 측정 목적에 맞아야 한다. 이 단계의 비교는 어느 도구가 본질적으로 더 빠른지를 재는 공정한 벤치마크가 아니라, Phase 0의 입력 부하를 만들 도구를 찾는 과정이었다.

두 도구의 강점은 다르다. Artillery는 VU별 행동, 사용자 대기시간(think time), 단계별 시나리오, 동적 페이로드처럼 실제 사용자 흐름을 표현하기 좋다. oha는 기능을 줄이는 대신 HTTP 요청을 빠르게 반복하는 데 집중한다.

사용자 행동을 재현하는 실험에서는 대기시간과 VU 모델도 측정 대상이다. 그러나 Phase 0의 목적은 collector 없이 ALB 입구의 처리 한계를 확인하는 것이었다. 생성 도구가 시나리오를 실행하느라 먼저 포화되면 ALB의 한계에 도달하지 못한다.

생성기 용량을 32 vCPU에서 16 vCPU로 줄이는 동시에 요청 본문을 오프라인에서 완성하고 제한 시간을 5초에서 10초로 늘렸다. 937만 5천 개 요청은 오류 없이 끝났지만 전체 처리량은 31.9k RPS, 가장 좋은 ALB 60초 구간도 33.5k RPS였다. 여러 조건이 함께 바뀌었으므로 32 vCPU 실행과의 차이를 CPU 축소 하나로 설명할 수는 없다. 확인된 값은 이 16 vCPU Artillery 구성의 처리량이다.

oha 비교에서는 16 vCPU, 연결 12,500개, 요청 수, HTTP 메서드와 경로, 제한 시간, keep-alive, 1,476바이트 요청 본문 크기를 맞췄다. 그러나 oha에는 Artillery의 0.15초 사용자 대기시간이 없었고 요청 본문 하나만 반복했다. oha는 같은 요청 수를 16.38초에 처리해 전체 실행시간 기준으로 17.95배 차이를 보였다. 이어진 75초 실행에서는 완전한 ALB 60초 구간에서 595.7k RPS와 활성 연결 12,500개를 기록했다. 종료 시점에 요청 10,622개가 중단됐지만 유효 성공률은 99.976%로 완료 기준을 통과했다. 이 실행은 지속 처리량 증거지만 오류 0의 최종 증거는 아니었다.

17.95배는 도구 자체의 효율 배수가 아니다. 사용자 대기시간과 요청 본문 선택 방식이 다른 두 부하 구성의 전체 실행시간 비율이다. 도구 성능만 비교하려면 대기시간, 페이로드 선택, 요청 속도와 종료 조건까지 같게 맞춰야 한다.

비교의 목적에는 충분했다. oha 구성은 필요한 50k RPS보다 훨씬 큰 입력 부하를 만들었고 별도의 75초 실행으로 60초 지속 처리량도 검증했다. 이후 가변 요청 본문과 고정 QPS를 추가해 Phase 0 완료 조건에 맞는 생성기로 조정했다. Artillery는 VU 행동과 대기시간을 표현해야 하는 실험에 계속 사용한다.

Phase 0의 처리 한계는 oha로 측정하되, 실제 사용자 시나리오가 필요한 단계에서는 Artillery를 쓸 수 있다. 하나의 표준 도구를 정하는 문제가 아니라 서로 다른 질문에 맞는 도구를 고르는 문제다.

## 무제한 발사는 안정성 증명에 부족했다

가능한 한 빨리 보내는 무제한 실행은 파괴 지점을 찾는 데 유용하다. 하지만 목표 RPS를 안정적으로 유지하는 능력은 보여주지 못한다. 크기가 다른 요청 본문은 현실성을 높이기 위한 조건이었고 안정성 기준을 통과한 실행에는 사전 예열과 고정 QPS를 함께 적용했다. 두 변경의 개별 효과는 이 실험에서 분리하지 못했다.

고정 요청 본문 하나만 반복하면 크기에 따른 직렬화 비용, 패킷 크기, 전송 바이트 수의 차이가 사라진다. Phase 0에서는 collector 로직을 실행하지 않지만 생성기와 네트워크가 실제 SDK 요청과 비슷한 크기를 다루게 할 필요는 있었다. 그래서 검증을 마친 요청 본문 여러 개를 미리 준비하고 요청마다 하나를 고르게 했다. 요청 시점에 복잡한 JSON을 다시 만들지 않으면서 크기 변화는 남길 수 있는 방식이다.

초기 oha 비교에서는 1,476바이트 요청 본문 하나를 반복했다. 실제 요청의 크기 변화를 살리기 위해 `compact`, `standard`, `expanded` 프로필에서 4개씩 골라 NDJSON 요청 본문 12개를 준비했다. oha `-Z` 옵션은 요청마다 한 줄을 무작위로 선택했다. 요청 본문 크기는 1,047~1,526바이트였다.

그다음 생성기 용량을 16 vCPU에서 2 vCPU로 줄였다. `c6in.large` 한 대가 무제한으로 보낸 실행은 55.7k RPS와 활성 연결 10,001개를 만들었지만 성공률은 99.43%였다. 목표 처리량은 넘었어도 시간 초과 24,554개가 발생해 완료 기준에는 미달했다. CPU 평균은 51.17%였으므로 단순한 CPU 부족으로 보기도 어려웠다.

무제한 생성기는 응답이 느려져도 가능한 한 다음 요청을 계속 시작한다. 지연시간이 늘어 처리 중인 요청이 많아지면 소켓과 버퍼가 더 오래 점유된다. 이 상태가 다시 지연시간을 늘리는 악순환으로 이어질 수 있다. 2 vCPU 실행에서 이런 현상과 일치하는 지연시간 증가와 시간 초과를 관측했지만 무제한 발사가 유일한 원인이라고 확인한 것은 아니다.

연결 10,000개를 두 `c6in.large`로 나누면 안정될 것으로 예상했지만 결과는 더 나빴다. 기존 생성기 경로는 56.7k RPS를 냈지만 새로 추가한 생성기 경로는 3.2k RPS와 71.21% 성공률에 그쳤다. 새 호스트의 CPU 사용률은 2.96%, ALB 활성 연결 수는 5,053이었다. 두 호스트 사이에 큰 비대칭이 있었다는 사실은 분명하지만 원인을 직접 측정한 값은 없었다.

보고서의 기존 경로와 새 경로라는 구분은 생성기 투입 순서를 가리킨다. 특정 ALB 노드의 준비 상태를 측정한 명칭이 아니다. 낮은 CPU 사용률과 활성 연결 수는 새 호스트가 계산을 충분히 수행하지 못했다는 해석을 뒷받침하지만 지연이 DNS 조회, TCP 핸드셰이크, 라우팅, ALB 연결 수립 중 어디에서 생겼는지는 확인하지 못했다.

예열은 측정 전에 낮은 부하를 보내 DNS 조회와 TCP 연결 수립을 마치고 커널 소켓 상태와 ALB까지의 경로를 활성화하는 과정이다. 연결 10,000개를 측정 시작 순간에 한꺼번에 만들 때 생기는 동시 집중도 측정 구간 밖으로 옮길 수 있다. 예열 구간은 실제 측정 결과에 포함하지 않았다.

고정 QPS 제한은 생성기가 목표보다 빠르게 요청을 늘리지 못하게 한다. 이 실험에서 알고 싶은 값은 무제한 최고점이 아니라 50k 이상을 오류 없이 유지하는지였다. 목표보다 여유 있는 60k를 일정하게 보내는 편이 질문에 더 정확했다.

인스턴스는 더 추가하지 않고 두 가지를 바꿨다.

- 측정 전에 각 경로에 낮은 QPS로 예열을 수행했다.
- 측정 중에는 oha `-q`로 호스트당 30k, 합계 60k RPS를 명시적으로 제한했다.

이 실행은 8,999,961개 요청을 오류 없이 처리했고 ALB 완전 구간은 60,002 RPS였다. CPU 평균은 13.57%에 불과했다. 무제한 최대치보다 목표 처리량을 안정적으로 유지하는 것이 Phase 0 완료 조건에 더 맞는 방식이었다.

인스턴스 추가만으로는 시간 초과를 해결하지 못했다. 같은 4 vCPU에서 예열과 `-q` 제한을 함께 적용하자 시간 초과가 91,799개에서 0개로 줄었다. 이 비교는 두 변경을 묶은 구성의 효과를 보여주며 예열과 QPS 제한 중 어느 쪽이 얼마나 기여했는지는 알려주지 않는다.

## 최종 구성은 2 vCPU와 55k RPS 제한

최종 생성기는 2 vCPU 한 대면 충분했다. 4 vCPU 성공 실행에서 CPU 여유를 확인한 뒤 한 대로 줄였고 55k RPS를 150초 동안 오류 없이 유지해 Phase 0을 마쳤다.

60k 성공 실행의 CPU 평균이 13.57%였다는 것은 두 생성기 모두 필요하지 않을 가능성을 보여줬다. 다만 CPU 사용률만 보고 바로 결론 내리지 않고 실제로 한 대를 제거한 뒤 같은 연결 수와 목표 이상의 QPS를 다시 측정했다. 측정 전 30초 동안 2,000개 연결과 20k RPS로 경로를 준비한 뒤, 10,000개 연결과 55k RPS 제한으로 150초 동안 측정했다.

이 실행은 생성기 수를 2대에서 1대로 줄이면서 제한값도 60k에서 55k RPS로 낮췄다. 따라서 2 vCPU가 이전과 같은 60k 부하를 처리한다는 비교는 아니다. Phase 0의 완료 기준인 50k RPS보다 10% 높은 부하를 오류 없이 유지했으므로 목표에 필요한 생성기 용량이 2 vCPU면 충분하다는 점을 검증한 것이다.

최종 결과는 다음과 같다.

- 생성기: `54,948.88 RPS`
- 완료 요청: `8,249,985 / 8,249,985`, 오류 0
- ALB 완전 60초 구간: `54,999.08 RPS`
- ALB `ActiveConnectionCount`: `15,233`
- 생성기/ALB 건수 차이: 5개, `0.0000606%`
- 지연시간: p50 `1.51ms`, p95 `3.28ms`, p99 `35.02ms`
- 생성기 CPU: 평균 `42.86%`, 최대 구간 `48.10%`
- ALB 연결 거부와 ELB 5xx: 없음

이 구성은 16 vCPU 비교 구성보다 vCPU를 87.5% 줄이면서 모든 완료 기준을 통과했다. 마지막 네 번의 용량 축소 실행과 공유 인프라 수명에 든 비용은 20% 여유분을 포함해 약 `$0.6080`이었다. 실험용 스택, ALB, ASG, EC2, EBS, VPC, IAM 정책, S3 접두사와 현재 실행의 로그 그룹도 모두 제거했다.

55k를 선택한 이유는 목표 50k에 10% 여유를 두기 위해서다. 정확히 50k로 제한하면 작은 스케줄링 흔들림만으로 완전한 60초 구간이 기준 아래로 내려갈 수 있다. 55k에서 오류가 없고 CPU 사용률도 절반 이하였으므로, 50k 목표를 위한 생성기 용량으로 충분하다고 판단했다.

`c6in.large` 한 대는 크기가 다른 SDK 호환 요청 본문, keep-alive 연결 10,000개, 55k 고정 QPS 조건에서 생성기와 ALB의 요청 건수가 사실상 일치하는 트래픽을 만들었다. 같은 생성 방식을 Phase 1에 쓰면 새로 추가되는 collector와 Kafka의 영향을 분리해서 볼 수 있다.

## Phase 0에서 확인한 원칙

성능 실험에서는 의도한 부하가 실제로 서버까지 도달했는지 확인한 뒤 서버가 느린 이유를 찾는다. Phase 0에서 얻은 원칙은 다음 단계에도 그대로 적용된다.

1. 트래픽이 시작되지 않은 계정 할당량과 용량 부족 실패는 서비스 성능 수치가 아니다.
2. 생성기 요청 수와 ALB 수신 수를 함께 봐야 생성기 병목을 서비스 병목으로 오인하지 않는다.
3. 낮은 CPU 사용률은 자원이 남았다는 증거일 수도 있지만 단일 이벤트 루프 포화처럼 병렬화되지 않는 병목의 신호일 수도 있다.
4. 페이로드 생성 비용, 연결 교체, 출발지 분산, 프로세스 수는 서로 다른 변수다. 하나씩 바꾸지 못했다면 결과를 개별 원인이 아니라 구성 단위로 해석한다.
5. 처리 한계 측정 도구와 사용자 행동 모델링 도구는 목적이 다르다. Phase 0의 한계 측정에는 oha, 행동 모델에는 Artillery가 더 적합하다.
6. 오류 없이 통과한 두 최종 구성은 예열과 고정 QPS를 함께 사용했다. 이번 실행만으로 두 조건의 개별 기여도는 분리하지 못했다.
7. 목표를 통과한 뒤 CPU와 오류율을 보고 다시 축소해 최종 용량을 결정한다.

ALB 고정 응답 경로는 최소 55k RPS를 안정적으로 처리했고 2 vCPU 생성기는 그 부하를 오류 없이 만들었다. Phase 1부터 같은 생성 방식을 사용하면 새로 나타나는 처리량 저하와 지연시간의 원인을 collector 또는 Kafka 경로에서 조사한다.

이 원칙은 특정 AWS 서비스에만 해당하지 않는다. 부하 생성기, 네트워크, 대상 서비스가 차례로 연결된 성능 실험에서는 앞단이 목표 부하를 만들지 못하면 뒷단의 처리 한계를 측정하지 못한다. 앞단과 대상의 요청 건수를 먼저 맞추고 다음 구성 요소를 하나씩 추가하면 병목 위치가 좁혀진다.

Phase 0의 핵심 산출물은 55k라는 숫자 하나가 아니다. 이후 단계에서 재사용 가능한 생성 방식과 병목 판정 방법까지 검증했다.

실행 방법과 산출물 규칙은 [AWS 이벤트 파이프라인 성능 테스트 가이드](../../reference/docs/guides/guide_aws_event_pipeline_performance_test.md)와 Phase 0 README (external snapshot reference: `../performance-tests/phase0/README.md`)에 분리되어 있다.
